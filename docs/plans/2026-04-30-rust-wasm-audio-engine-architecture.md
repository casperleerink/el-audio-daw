# Rust/WASM Browser Audio Engine Architecture

## Context

This project is experimental/throwaway. There are no production users and no compatibility requirements with the existing Elementary Audio implementation.

Therefore, the goal is **not** to preserve the current `packages/audio` API or migrate incrementally around old abstractions. The goal is to design the browser Rust audio engine cleanly from scratch, while keeping the broader product direction:

```txt
Browser DAW
React UI
Zero Sync for realtime collaboration/project state
Rust/WASM AudioWorklet for realtime audio
```

## Core decision

Build a new browser audio engine around a Rust/WASM `AudioWorklet`.

Recommended architecture:

```txt
Zero Sync project state
  -> React project adapter
  -> Audio engine controller
  -> AudioWorkletNode
  -> Rust/WASM realtime engine
  -> Web Audio destination
```

Zero remains the collaborative source of truth. The Rust engine is a local deterministic renderer of the current project state.

## Important principle

Do **not** preserve Elementary-era concepts unless they are genuinely useful.

Avoid carrying over APIs such as:

```ts
setTracks(tracks);
setClips(clips);
setEffects(effects);
loadAudioIntoVFS(assetId, url);
```

These were shaped by the old implementation. In particular, `loadAudioIntoVFS` is Elementary-specific and should not survive.

Instead, design the new engine around:

- project snapshots,
- explicit engine commands,
- asset loading,
- transport control,
- realtime-safe parameter changes,
- future automation/plugin support.

## Target high-level data flow

```txt
Zero query result
  -> derive ProjectAudioState
  -> AudioEngineController.loadProject() / applyProjectUpdate()
  -> postMessage / command queue
  -> AudioWorkletProcessor
  -> Rust engine
  -> audio output
```

The UI should not manipulate low-level DSP nodes. React/Zero should produce durable project state. The audio engine should decide how to render it.

## Package direction

The existing `packages/audio` package can remain as the browser audio package, but its internals and public API may be rewritten.

Suggested layout:

```txt
packages/audio
  src/
    index.ts
    controller.ts          # main-thread AudioEngineController
    protocol.ts            # command/event DTOs
    project-audio-state.ts # clean audio DTOs derived from Zero
    assets.ts              # browser asset decode/load helpers
    worklet/
      processor.ts         # AudioWorkletProcessor glue
  rust/
    Cargo.toml
    src/
      lib.rs
      engine.rs
      transport.rs
      timeline.rs
      graph.rs
      mixer.rs
      assets.rs
      automation.rs
      dsp/
        mod.rs
        gain.rs
        pan.rs
        filter.rs
  pkg/                     # generated by wasm-pack
```

This can be changed as implementation teaches us more, but the key idea is:

```txt
TypeScript = browser integration and command transport
Rust = audio semantics and DSP
```

## Recommended public TypeScript API

Design a new API instead of preserving the old one.

```ts
export class AudioEngineController {
  initialize(options: AudioEngineOptions): Promise<void>;
  dispose(): void;

  loadProject(project: ProjectAudioState): void;
  applyProjectUpdate(update: ProjectAudioUpdate): void;

  loadAsset(asset: AudioAssetSource): Promise<void>;
  unloadAsset(assetId: string): void;

  play(): void;
  pause(): void;
  stop(): void;
  seek(timeSeconds: number): void;

  setTrackGain(trackId: string, gainDb: number): void;
  setTrackPan(trackId: string, pan: number): void;
  setMasterGain(gainDb: number): void;

  onPlayhead(callback: (event: PlayheadEvent) => void): () => void;
  onMeters(callback: (event: MeterEvent) => void): () => void;
  onError(callback: (event: AudioEngineError) => void): () => void;
}
```

The controller is a browser/main-thread object. It owns:

- `AudioContext`,
- `AudioWorkletNode`,
- worklet module loading,
- asset fetching/decoding,
- command/event serialization,
- ergonomic API for React/Zustand.

It should not contain core audio logic.

## Project audio state DTO

Introduce one canonical audio snapshot derived from Zero.

```ts
export type ProjectAudioState = {
  projectId: string;
  sampleRate: number;
  tempoMap: TempoMapState;
  tracks: TrackAudioState[];
  clips: ClipAudioState[];
  routes: RouteAudioState[];
  effects: EffectAudioState[];
  automation: AutomationLaneState[];
  master: MasterAudioState;
};
```

Initial simplified shape can be smaller:

```ts
export type ProjectAudioState = {
  projectId: string;
  sampleRate: number;
  tracks: TrackAudioState[];
  clips: ClipAudioState[];
  effects: EffectAudioState[];
  master: {
    gainDb: number;
  };
};
```

Track:

```ts
export type TrackAudioState = {
  id: string;
  kind: "audio" | "instrument" | "bus" | "master";
  order: number;
  muted: boolean;
  solo: boolean;
  gainDb: number;
  pan: number;
};
```

Clip:

```ts
export type ClipAudioState = {
  id: string;
  trackId: string;
  assetId: string;
  startSamples: number;
  durationSamples: number;
  sourceStartSamples: number;
  gainDb: number;
  fadeInSamples?: number;
  fadeOutSamples?: number;

```

Effect:

```ts
export type EffectAudioState = {
  id: string;
  ownerType: "track" | "bus" | "master";
  ownerId: string;
  order: number;
  enabled: boolean;
  plugin: InternalPluginState | ExternalPluginState;

```

This DTO is independent from Zero table shapes. Zero can change without contaminating engine internals.

## Command protocol

The controller sends commands to the worklet.

Start with structured `postMessage` commands for simplicity:

```ts
export type EngineCommand =
  | { type: "LoadProject"; project: ProjectAudioState }
  | { type: "ApplyProjectUpdate"; update: ProjectAudioUpdate }
  | { type: "LoadAsset"; asset: DecodedAudioAsset }
  | { type: "UnloadAsset"; assetId: string }
  | { type: "Play" }
  | { type: "Pause" }
  | { type: "Stop" }
  | { type: "Seek"; timeSeconds: number }
  | { type: "SetTrackGain"; trackId: string; gainDb: number }
  | { type: "SetTrackPan"; trackId: string; pan: number }
  | { type: "SetMasterGain"; gainDb: number };
```

Later, if profiling shows structured cloning/JSON/object allocation is a problem, move high-frequency commands to:

```txt
SharedArrayBuffer ring buffer
or binary command buffer
```

Do not optimize this before the architecture works.

## Rust engine model

Rust should own the audio domain model optimized for rendering, not mirror React or Zero directly.

Suggested Rust shape:

```rust
pub struct Engine {
    sample_rate: f32,
    transport: Transport,
    timeline: Timeline,
    graph: AudioGraph,
    assets: AssetStore,
    meters: MeterState,
    scratch: ScratchBuffers,
}
```

Transport:

```rust
pub struct Transport {
    state: TransportState,
    position_samples: u64,
    loop_region: Option<LoopRegion>,
}

pub enum TransportState {
    Stopped,
    Playing,
    Paused,
}
```

Timeline:

```rust
pub struct Timeline {
    tracks: Vec<Track>,
    clips: Vec<Clip>,
    tempo_map: TempoMap,
    automation: AutomationStore,
}
```

Graph:

```rust
pub struct AudioGraph {
    track_nodes: Vec<TrackNode>,
    master: MasterNode,
}
```

Processing entry point:

```rust
impl Engine {
    pub fn process(&mut self, left: &mut [f32], right: &mut [f32]) {
        self.clear_outputs(left, right);

        if !self.transport.is_playing() {
            self.update_meters(left, right);
            return;
        }

        self.render_timeline_block();
        self.process_graph();
        self.write_master_output(left, right);
        self.transport.advance(left.len() as u32);
    }
}
```

## Build pipeline recommendation

Use **`wasm-pack` with `wasm-bindgen` and `--target web`** for the first implementation.

Recommended initial pipeline:

```txt
Rust crate
  -> wasm-pack build --target web
  -> generated JS/WASM package
  -> Vite imports generated JS/WASM URL
  -> AudioWorkletProcessor initializes WASM
```

Do not start with raw `wasm32-unknown-unknown`. Raw WASM may be useful later, but it adds unnecessary early friction:

- manual memory management,
- manual imports/exports,
- harder TypeScript integration,
- more AudioWorklet boilerplate,
- harder Vite setup.

Use `wasm-pack` first to prove the engine architecture. Optimize later only if profiling shows generated glue is a real problem.

Suggested build command:

```bash
wasm-pack build packages/audio/rust --target web --out-dir ../pkg
```

Suggested package script:

```json
{
  "scripts": {
    "build:wasm": "wasm-pack build rust --target web --out-dir ../pkg",
    "check-types": "tsc --noEmit"
  }
}
```

Use explicit WASM URL loading where possible:

```ts
import init, { WasmEngine } from "../../pkg/audio_engine.js";
import wasmUrl from "../../pkg/audio_engine_bg.wasm?url";

await init(wasmUrl);
const engine = new WasmEngine(sampleRate);
```

This is usually less surprising inside AudioWorklet/Vite module loading than relying on implicit path resolution.

## AudioWorklet shape

Main thread controller:

```ts
await audioContext.audioWorklet.addModule(
  new URL("./worklet/processor.ts", import.meta.url),
);

const node = new AudioWorkletNode(audioContext, "daw-audio-processor", {
  numberOfInputs: 0,
  numberOfOutputs: 1,
  outputChannelCount: [2],
});

node.port.postMessage({ type: "Initialize", sampleRate: audioContext.sampleRate });
node.connect(audioContext.destination);
```

Processor:

```ts
class DawAudioProcessor extends AudioWorkletProcessor {
  private engine: WasmEngine | null = null;

  constructor() {
    super();

    init(wasmUrl).then(() => {
      this.engine = new WasmEngine(sampleRate);
    });

    this.port.onmessage = (event) => {
      this.engine?.applyCommand(event.data);
    };
  }

  process(_inputs: Float32Array[][], outputs: Float32Array[][]) {
    const output = outputs[0];
    const left = output?.[0];
    const right = output?.[1];

    if (!this.engine || !left || !right) {
      return true;
    }

    this.engine.process(left, right);
    return true;
  }
}

registerProcessor("daw-audio-processor", DawAudioProcessor);
```

Avoid hot-path allocation.

Good:

```ts
engine.process(left, right);
```

Bad:

```ts
const rendered = engine.process();
left.set(rendered.left);
right.set(rendered.right);
```

The render callback runs every 128 samples. It must be boring, predictable, and allocation-light.

## Initial Rust wasm-bindgen API

Start simple:

```rust
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub struct WasmEngine {
    engine: Engine,
}

#[wasm_bindgen]
impl WasmEngine {
    #[wasm_bindgen(constructor)]
    pub fn new(sample_rate: f32) -> Self {
        Self {
            engine: Engine::new(sample_rate),
        }
    }

    pub fn process(&mut self, left: &mut [f32], right: &mut [f32]) {
        self.engine.process(left, right);
    }

    pub fn apply_command(&mut self, command: JsValue) {
        self.engine.apply_command(command);
    }
}
```

This is acceptable for the first prototype. Later, replace `JsValue` command handling with typed/binary commands if needed.

## Asset loading

Do not use the old VFS language.

Target asset flow:

```txt
asset reference from Zero
  -> browser fetches URL
  -> AudioContext.decodeAudioData
  -> extract Float32Array channel data
  -> send DecodedAudioAsset to worklet
  -> Rust stores decoded PCM in AssetStore by assetId
```

TypeScript source object:

```ts
export type AudioAssetSource = {
  id: string;
  url: string;
};
```

Decoded payload:

```ts
export type DecodedAudioAsset = {
  id: string;
  sampleRate: number;
  channels: Float32Array[];
  lengthSamples: number;
};
```

For MVP, full decoded buffers in memory are fine. Later, serious sessions need:

- chunked loading,
- streaming decode,
- cache eviction,
- memory budgeting,
- background waveform generation.

Zero should store asset metadata/references only, never raw audio binary.

## Timeline units

The current project stores clip positions in samples. That can continue initially, but define the meaning precisely.

There are three sample-rate domains:

```txt
project timeline sample rate
AudioContext render sample rate
source asset sample rate
```

The engine must own conversion between these domains.

Recommended initial rule:

```txt
Project timeline samples are measured at ProjectAudioState.sampleRate.
Render samples are measured at AudioContext.sampleRate.
Asset source samples are measured at asset.sampleRate.
```

Long-term, if tempo/musical editing becomes central, consider ticks/beats for musical positions. For an audio-first DAW, project samples are acceptable if conversion is centralized.

## Automation

Automation should be designed as first-class data, not as UI-driven parameter spam.

Future Zero tables:

```txt
automation_lanes
automation_points
```

DTO shape:

```ts
export type AutomationLaneState = {
  id: string;
  target: AutomationTarget;
  points: AutomationPointState[];
};

export type AutomationTarget =
  | { type: "track"; trackId: string; param: "gain" | "pan" }
  | { type: "effect"; effectId: string; param: string }
  | { type: "master"; param: "gain" };

export type AutomationPointState = {
  timeSamples: number;
  value: number;
  curve: "step" | "linear" | "exponential";
};
```

Rust should evaluate automation during rendering. For gain/pan/filter cutoff, this should eventually be sample-accurate or at least generated as per-sample/per-block parameter buffers.

## Realtime safety

The AudioWorklet render path should avoid:

- network access,
- file IO,
- locks,
- unbounded allocation,
- JSON parsing in the audio callback,
- expensive graph rebuilds,
- logging from the hot path.

For the prototype, some compromises are acceptable outside `process()`. The `process()` method itself should stay disciplined from day one.

## Metering and playhead events

The Rust engine should produce metering/playhead data. The worklet sends events to the controller.

Start with periodic `postMessage` events:

```ts
export type EngineEvent =
  | { type: "Ready" }
  | { type: "Playhead"; timeSeconds: number; positionSamples: number }
  | { type: "Meters"; meters: MeterValue[] }
  | { type: "Error"; message: string };
```

Later, move high-frequency meter/playhead data to `SharedArrayBuffer` if needed.

Note: `SharedArrayBuffer` requires cross-origin isolation headers.

## WAM integration stance

A Rust/WASM engine does not automatically make WAM per-track inserts easy.

WAM plugins are normally separate `AudioWorkletNode`s in the Web Audio graph. If the whole DAW mixer lives inside one Rust worklet, inserting WAM plugins inside individual track chains is not trivial.

Recommended approach:

### Phase 1

Build one Rust/WASM worklet engine:

```txt
Rust/WASM Engine AudioWorklet
  -> destination
```

This gives the best control over:

- clips,
- transport,
- automation,
- mixer behavior,
- deterministic rendering.

### Phase 2

Build internal Rust effects first:

- gain,
- pan,
- filter,
- EQ,
- compressor,
- delay/reverb later.

### Phase 3

Add WAM where the browser graph makes sense:

```txt
Rust engine output
  -> WAM master effects
  -> destination
```

Later, if WAM per-track inserts become essential, revisit topology:

```txt
Rust track/stem outputs
  -> WAM chains
  -> Web Audio or Rust master bus
  -> destination
```

Do not make WAM the foundation of the first rewrite.

## Suggested implementation phases

### Phase 1: New package skeleton

- Rewrite `packages/audio` around `AudioEngineController`.
- Add `protocol.ts` and `project-audio-state.ts`.
- Add Rust crate under `packages/audio/rust`.
- Add `wasm-pack` build script.
- Add AudioWorklet processor.

Success condition:

```txt
Browser initializes AudioContext + worklet + WASM.
Worklet outputs silence or sine wave.
```

### Phase 2: Transport

- Implement play/pause/stop/seek.
- Maintain sample-accurate transport position in Rust.
- Emit playhead events.

Success condition:

```txt
UI can start/stop/seek and receive playhead updates from the worklet.
```

### Phase 3: Assets and clip playback

- Fetch/decode audio assets in the controller.
- Send decoded PCM to the worklet.
- Store assets in Rust.
- Render clips at timeline positions.
- Support clip source offset and duration.

Success condition:

```txt
Existing project clips can play back from Rust without Elementary.
```

### Phase 4: Tracks and mixer

- Track summing.
- Mute/solo.
- Track gain.
- Constant-power pan.
- Master gain.
- Basic meters.

Success condition:

```txt
Current DAW mixer features work through the Rust engine.
```

### Phase 5: Internal effects

- Implement current filter effect in Rust.
- Add a small internal plugin/node trait.

Success condition:

```txt
Track effect chain works without Elementary.
```

### Phase 6: Automation

- Add automation DTOs.
- Add Zero schema later.
- Implement gain/pan automation first.

Success condition:

```txt
Track gain or pan automation renders from stored automation points.
```

### Phase 7: WAM experiments

- Try master insert WAM chain after Rust output.
- Measure complexity before attempting per-track WAM inserts.

## When to move beyond wasm-pack

Stay with `wasm-pack` until there is a measured reason to change.

Consider raw `wasm32-unknown-unknown` or a custom build if:

- generated JS glue causes AudioWorklet issues,
- binding overhead is measurable,
- hot-path allocations cannot be eliminated,
- binary size matters,
- shared memory/ring-buffer integration needs tighter control.

Do not optimize this before the Rust engine proves the architecture.

## Summary

Because this is throwaway and there are no compatibility constraints, do not preserve the existing Elementary-shaped API.

Recommended path:

```txt
Browser first
React + Zero remain product foundation
Rewrite packages/audio cleanly
New project/command-oriented AudioEngineController
Rust/WASM AudioWorklet backend
wasm-pack + wasm-bindgen --target web initially
Internal Rust DSP/effects first
WAM later as an integration layer
```

This gives the cleanest path toward serious DAW features while keeping the browser and Zero collaboration model.
